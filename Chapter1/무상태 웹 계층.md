## 무상태 웹 계층

- 사용자 세션 데이터와 같은 상태 정보를 웹 계층에서 제거하는 것
- 웹 계층을 수평적으로 확장하기 위해 필요
    - 상태를 가지게 되면, 수평적 확장 시에 상태를 각 서버가 공유해야할 필요가 생기기 때문
- 상태를 데이터베이스에 보관하고, 필요 시 가져오는 전략을 사용함

### 상태 정보 의존적인 아키텍처

- 상태 정보를 저장하는 구조에서는, 같은 클라이언트의 요청은 반드시 같은 서버로 전송되어야 하는 단점이 있음
- 이를 가능하게 하기 위해서 고정 세션이란 기술을 사용함.
    - **Q. 고정 세션이란?**
        - 로드 밸런서가 특정 클라이언트의 요청을 항상 동일한 서버로 라우팅하도록 보장하는 기술
        - 클라이언트 요청이 처음 들어오면, 로드밸런서는 클라이언트와 특정 서버를 묶는 식별자(쿠키 기반 or IP 해시 기반)를 생성하고, 이를 통해 라우팅함
            - 쿠키 기반1 (AWS ALB, Nginx)
                - 로드밸런서가 클라이언트로 응답 내릴 때 특수 쿠키를 포함해서 내림
                    - e.g. `Set-Cookie: LB_ROUTE=server1`
                - 이후 클라이언트가 요청을 보낼 때 해당 쿠키를 포함해 보냄
                - 로드밸런서는 쿠키를 판독하여 라우팅
                - 단점
                    - LB 가 쿠키를 볼 수 있어야 동작 가능
            - 쿠키 기반2
                - 서버가 발행하는 세션 쿠키를 LB 가 감지하여 클라이언트와 매핑
                - 클라이언트가 세션 쿠키를 포함해 요청을 보냄
                - 로드밸런서는 쿠키 판독해 매핑된 서버로 요청을 라우팅
                - 단점
                    - 매핑 테이블을 저장할 추가적인 메모리 필요함
                    - LB 를 여러 대 사용하는 경우, 매핑 테이블의 싱크를 맞춰야 함
            - IP 해시 기반
                - 클라이언트의 IP 주소를 해시해서 서버를 결정
                - 여러 사용자가 같은 공인 IP 를 사용하는 NAT 환경에서는 특정 서버에 부하가 몰릴 수 있음
                - 모바일 환경처럼 IP 가 자주 변경되는 경우에는 사용할 수 없음
