## 메시지 큐

- 메시지 무손실을 보장하는, 비동기 통신을 지원하는 컴포넌트
    - 엄밀히 말하자면, 무손실 보장은 큐 전략 설정에 따라 달라짐
    - 내구성 있는 메시지 전달을 지원한다고 이해하는 게 더 적절함
- 메시지 버퍼 역할을 하며, 비동기적으로 메시지를 전송함
- 기본 아키텍처는 생산자(발행자) 가 메시지를 만들어 발행하면, 큐에 붙은 소비자(구독자) 가 메시지를 받아 동작을 수행하는 구조
- 메시지 큐를 사용하면 서버 간의 결합을 느슨하게 만들 수 있음
    - 때문에 규모 확장성이 보장되어야 하는 애플리케이션을 구성하기 좋음
- **Q. 큐가 다 차면 어케 됨? 전략이 다양할 거 같은데**
    1. Drop Tail
        1. 새롭게 들어온 메시지를 그냥 버리는 방식
        2. 일부 메시지 유실이 허용되는 경우 사용함 (e.g. 로그 수집)
        3. 구현이 단순함
        4. 메시지가 유실될 수 있음
    2. Drop Head
        1. 큐에서 가장 오래된 메시지를 버리는 방식
        2. 최신성이 중요한 경우에 사용함 (e.g. 실시간 상태 업데이트)
    3. Blocking (생산자 대기)
        1. 새 메시지를 바로 큐에 못하게 하고, 공간이 생길 때까지 생산자 쪽에서 잠깐 기다리게 하는 방식
            1. 보통 일정 시간까지만 대기하게 하고 넘어가면 실패하게 하는 방식으로 사용함
        2. 생산된 메시지를 반드시 처리해야 하는 경우에 사용 (e.g. 내부 시스템)
        3. 메시지 유실 가능성이 적음
        4. 그러나 생산자 응답시간이 길어지고, 블로킹이 심하면 전체 서비스가 지연됨
    4. 즉시 실패 반환
        1. 메시지가 들어올 공간이 없으면, 요청 자체를 거절하는 방식
        2. 가장 보수적인 방식
        3. 그러나 별도의 재시도 로직을 구현해야함
- **추가 개념!**
    - Persistent
        - 메시지 유실을 방지하기 위해, 메시지를 메모리에만 저장하는 게 아니라 디스크에도 기록하는 방식
            - 처음부터 디스크 사용함
        - 프로세스 종료, 서버 재시작, 장애 발생 시에도 복구 가능
        - 그러나 디스크를 사용하므로 느림. 그리고 디스크 장애라는 리스크가 추가됨
            - 디스크 장애는 보통 replication 으로 완화함
    - Spill-to-disk
        - 평상시에는 메모리에서 빠르게 처리하다가, 메모리가 부족하면 넘치는 데이터를 디스크 용량을 사용해 처리하는 방식
            - 순간적인 버퍼 확장 개념으로 디스크 사용함
        - 순간적인 폭증에 대처가 가능하며, 메시지 유실 대신 지연으로 흡수 가능함
        - 얘도 디스크를 사용하므로 느림. 그리고 메시지 처리가 지연되면 디스크 용량도 다 찰 수 있음
            - 디스크 용량 다 차면, 얘도 결국 저 위의 전략 중 하나를 사용해야함
    - 결국 둘 다 완벽한 해결책은 아님. 병목 지점을 확인해서 적절한 추가 조치를 해야함
        - 예를 들어, 컨슈머 처리 속도가 느리다면 컨슈머를 추가하는 등의 조치가 동반되어야 함!
