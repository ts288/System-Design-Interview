## 어떤 DB 를 사용할 것인가?
### RDBMS

- 관계형 데이터베이스
- 자료를 열, 칼럼으로 표현
- join 가능

### NoSQL

- 종류를 4가지로 나눌 수 있음
    - 키-값 저장소 (Redis, DynamoDB)
        - 데이터를 키와 값으로 표현 (e.g. user:1001)
        - 키로 조회할 때 빠름
            - 해시 기반이면 보통 평균 O(1), 해시 충돌 등의 이유로 최악의 경우 O(n)
            - 트리 기반이면 O(log n)
        - 구조가 단순해서 캐시, 세션 저장, 설정값 저장에 적합
        - 값 내부를 조건 검색하는 건 느림
    - 그래프 저장소 (Neo4j, Amazon Neptune)
        - 노드, 엣지로 구성됨 (e.g. A --친구--> B)
        - 데이터 사이의 관계를 중심으로 저장
        - 소셜 네트워크 관계 분석, 추천 시스템, 지식 그래프 등에 적합
    - 칼럼 저장소 (Cassandra, HBase)
        - 칼럼 또는 칼럼 그룹 중심으로 데이터를 다룸 (e.g. 사용자 데이터 중, 이름 열만 모아 저장)
        - 대용량 데이터 분석에 적합
        - 데이터가 크고 분산 저장이 필요할 때 사용
            - 예를 들어서, 유저 데이터 중에서 나이만 분석해야 할 때, 전부 다 가져오면 비효율적
            - 그래서 칼럼 단위로 저장하면 원하는 데이터만 효율적으로 읽을 수 있음
        - 읽기 쓰기 패턴이 명확한 대규모 시스템에 적합
        - 로그/이벤트 저장, 대규모 분산 처리 등에 사용
        - 범위 스캔, 시간순 데이터 읽기, 쓰기 작업이 많은 경우에 적합
    - 문서 저장소 (MongoDB, Couchbase)
        - 데이터를 문서 단위로 저장(보통 json 구조를 그대로 저장)
        - 데이터 구조가 유연해, 객체 하나를 문서처럼 저장 가능
        - 스키마가 고정되지 않은 데이터에 적합
        - 카탈로그, 사용자 프로필, 콘텐츠 관리 시스템 등에 적합
- join 불가능

## 언제 NoSQL 이 유용한가?

- 낮은 지연시간이 요구될 때
- 다루는 데이터가 비정형일 때
- 데이터를 직렬화/역직렬화만 하면 될 때
- 많은 양의 데이터를 저장할 필요가 있을 때
