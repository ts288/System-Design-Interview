## CDN (콘텐츠 전송 네트워크)

- 정적 콘텐츠 전송에 쓰이는 분산된 서버의 네트워크
    - 여기서 ‘네트워크’ 라고 하는 것은 폐쇄망 개념이 아니라, 그냥 분산된 서버들의 집합을 말하는 것
- 이미지, 비디오, css, js 캐시 가능
- **Q. 동적 콘텐츠 캐싱은 머임?**
    - 요청 경로, 질의 문자열, 쿠키, 요청 헤더 기반해 html 페이지를 캐시하는 것
- **Q. 웹서버 ip 로 요청 들어오는데 어떻게 CDN 이 응답을 줌?**
    - 애초에 웹서버 ip 로 요청 보내는 게 아니라, CDN 한테 요청 보냄
    - 데이터 있으면 반환, 없으면 CDN 이 원본 서버에 데이터 요청하고, 받은 데이터 반환
        - 이 때 CDN 이 받는 데이터에는 TTL 이 걸려있음
        - 이 경우, 원본 서버는 웹서버일 수도 있고, s3 일 수도 있음

### CDN 사용 시 고려할 사항

- 비용
    - cdn 은 입출력 데이터 전송 양에 따라 요금을 내게 됨
    - 따라서, 최대한 안 써야 함
- TTL 설정
    - 너무 길면 업뎃이 늦고, 너무 짧으면 불필요한 업뎃이 됨
    - 이건 캐시 전략 참고하면 좋을 듯 함
        - 데이터의 중요도에 따라 전략 다르게 가져가게 설정!
- CDN 장애에 대한 대처
    - 원본 서버에서 직접 데이터를 가져오게 설정해야함
    - **Q. CDN 은 LB 에 의해 로드밸런싱이 되지 않는데, 얘가 죽었는지, 응답 안하는지는 클라이언트가 판단해야하나?**
        - ㄴㄴ DNS 가 해줌
        - CDN 이 자체적으로 헬스 체크를 제공해서, DNS 가 CDN 주소 반환할 때 **살아있는 서버 주소만 넘김**
- 콘텐츠 무효화 방법
    - CDN 서비스가 제공하는 API 사용
    - 오브젝트에 버전을 추가하여 질의 시 버전 필드 추가
- **Q. 가장 가까운 CDN 은 어떻게 찾음?**
    - CDN 은 분산된 서버들에 동일한 IP 를 부여함
    - 그럼에도 불구하고, 가장 가까운 서버로 라우팅될 수 있는 것은 BGP(Border Gateway Protocol) 덕분임
    - **BGP?**
        - 인터넷 장비들이 "나는 어느 IP 대역으로 가는 길을 알고 있어"를 서로 광고하는 프로토콜
        - 주변 기기들에게 ‘내가 어느 IP 대역으로 가는 길을 알고 있는지, path 어떻게 되는지’ 를 전파함
            - { destination: 1.1.1.1, AS-PATH: [Cloudflare] } 이런 형식으로 전달됨
            - 여기서 AS-PATH 는 상대 장비까지 도달하는 경로를 누적해둔 것임
            - 이 경로가 짧은 것을 우선적으로 선택함
        - 예를 들어, 서울과 뉴욕에 있는 CDN이 각각 주변 라우터에게 BGP 를 사용해 "나는 1.1.1.1을 가지고 있어"를 광고한다면, 광고가 라우터를 거칠수록 AS-PATH가 누적되어 길어짐. 따라서 SKT 라우터는 AS-PATH가 더 짧은 서울 CDN을 선택하게 되는 것!
