## 캐시

- 자주 사용하거나, 복잡한 연산 결과를 저장해두고 사용
- 이를 통해 비싼 연산을 줄일 수 잇음(e.g. db 호출)
- 애플리케이션 성능은 db 호출 횟수에 따라 크게 좌우되기 때문에, 캐시를 사용하면 성능을 향상시킬 수 있음

### 캐시 계층

- 데이터가 잠시 보관되는 곳. DB 보다 빠름
    - **Q. 빠르다는 걸 좀 더 자세히 정의하면?**
        - 저장 구조 관점에서, 메모리를 사용하기 때문에 빠름
        - 더 중요한 건 처리 복잡도가 DB 에 비해 월등히 적다는 것
            - DB 처리 시 고려 사항
                - SQL 파싱
                - 실행 계획 작성
                - 인덱스 탐색
                - MVCC/락/트랜잭션 처리
                    - MVCC 는 동시성 제어를 위해 사용하는 버전 컨트롤 방식
                - **즉, 데이터를 꺼내기 위해 많은 과정이 필요함**
            - 캐시 처리 시 고려 사항
                - key 로 바로 값을 찾고
                - **이미 가공된 값을 꺼내고**
                - 트랜잭션, 조인, 복잡한 실행계획을 거의 고려하지 않음

- **읽기 주도형 캐시 전략(Cache-Aside)**
    - 애플리케이션이 직접 캐시 관리
    - 읽을 때만 캐시 확인
    - 단순하나, 캐시 미스 시 DB 부하가 증가할 수 있음
    1. 사용자가 데이터 요청
    2. 캐시에 데이터가 있는지 확인
    3. 있다면 반환, 없다면 애플리케이션이 db 에서 가져와서 캐시에 저장하고 반환
- **Q. 다른 전략?**
    - **Read-Through**
        - 캐시가 DB 조회까지 대신 수행
        - Cache-Aside보다 추상화 수준이 높음
            - 애플리케이션은 캐시만 보고, 캐시가 내부적으로 DB 조회까지 처리하기 때문
        1. 사용자가 데이터 요청
        2. 캐시에 데이터가 있는지 확인
        3. 없다면 **캐시가** db 에서 데이터를 가져와서 캐시에 저장하고 반환
    - **Write-Through**
        - 데이터 쓰기 시점에 DB 랑 캐시에 동시에 반영
            - 캐시와 DB 가 다른 시스템일 경우, 하나의 트랜잭션으로 묶기 어려움
            - 그래서 DB 를 성공 기준으로 판단함
                - 아웃박스 패턴은 사용할 수 없음. 왜냐면 이 패턴을 여기에 적용하면 결국 DB 업데이트 -> 캐시 업데이트가 되기 때문에, 동시 반영이 되지 않기 때문
        - 항상 최신 데이터 유지할 수 있으나, 쓰기 성능이 저하됨
    - **Write-Behind**
        - 캐시에 먼저 쓰고, DB 에 비동기 반영
        - JPA 영속성 컨텍스트랑 비슷한 개념
            - JPA 영속성 컨텍스트에서는 1차 캐시에 저장하고 사용하다가 트랜잭션 커밋 시점에 DB 반영
        - Write-Through 에 비해 쓰기 성능은 개선됨
        - 장애 시 데이터 유실 가능성이 있고, 일관성 유지가 어려움
            - 멀티 인스턴스 구조에서, 모든 캐시가 동일한 정보를 가지지 않기 때문
            - DB 변경이 캐시에 바로 반영되지 않기 때문
    - **Refresh-Ahead**
        - 캐시에 TTL 을 걸고, **만료되기 전**에 미리 데이터를 갱신해두는 방식
        - 캐시 미스 시 DB 부하를 줄일 수 있음
        - 이건 조회가 매우 많거나, 캐시 미스가 치명적인 경우에만 사용함
            - 불필요한 DB 조회가 늘어날 수 있기 때문
    - **전략 고려 시에는 데이터 종류, 크기, 엑세스 패턴에 따라 구분해야함**

### 캐시 사용 시 유의할 점

- 어떤 경우에 사용해야하나?
    - 캐시는 데이터 참조는 빈번하지만, 갱신은 자주 이뤄지지 않는 경우에 적합
- 어떤 데이터를 두어야 하나?
    - 휘발성 메모리 사용하므로, 영속화 대상인 데이터를 두면 안돼
- 만료는 어떻게?
    - 정책을 따로 둬야 함.
    - 너무 짧거나 길면 db 조회를 효율적으로 하지 못하거나, 원본과 차이가 날 수 있음
    - **Q. 이건 어떻게 해결?**
        - 데이터 성격에 따라 다르게 두는 게 좋음
            - 일반 데이터 : TTL 설정 (시간은 데이터 변경에 따라 가변적으로 잡음)
            - 매우 중요한 데이터 : TTL 없이 쓰기 발생 시 캐시 갱신하는 전략 사용
- 일관성 유지?
    - 데이터 원본과 캐시 사본이 같게 해야하는데 어떻게?
    - **Scaling Memcache at Facebook 논문 참조, 이건 추후 정리**
- 장애 대처?
    - 캐시가 단일 장애 지점이 될 수 있음
    - 그래서 캐시를 분산시켜야함
- 캐시 크기는 얼마나 크게?
    - 과할당 하면 우선 공간 없어서 저장 못하는 일은 사라짐
    - **Q. 이게 최선인가?**
        - 무조건 크게 잡으면 비용 측면에서 낭비임. 안그래도 비싼데 ㅠ
        - 그래서 적중률 기준으로 잡는 게 맞음. 이건 모니터링으로 확인 (캐시 히트 수 / 전체 조회 수)
- 데이터 방출은 어떻게?
    - 캐시가 다 차면 필요없는 거 내보내야함. 이 때 전략이 필요해
        - LRU - 마지막 사용 시점이 가장 오래된 거 방출
        - LFU - 사용 빈도 가장 낮은 거 방출
        - FIFO - 젤 먼저 들어온 거 방출
