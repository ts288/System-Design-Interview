## 수직적 규모 확장 vs 수평적 규모 확장

### 수직적 규모 확장

- 서버에 고사양 자원을 추가하는 것
- 유입되는 트래픽이 적을 때 좋은 선택지
    - 서버를 추가하는 것보다 비용이 적게 들기 때문
- 단순함
- 그러나 한계가 명확함
    - 규모 확장에 한계가 있음
    - 단일 장애점(SPOF)이 됨

### 수평적 규모 확장

- 서버의 개수 자체를 늘리는 것
- 유입되는 트래픽이 많을 때 좋은 선택지
- 복잡함
- 단일 장애점 문제 해결 가능


### 로드밸런서

- 수평적 확장을 하는 경우, 유입되는 트래픽을 특정 서버로 보내는 역할을 수행할 필요가 있음
- 이걸 로드밸런서가 해줌
- 로드밸런서는 부하 분산 집합에 속한 서버들에게 트래픽 부하를 분산하는 역할을 함
- 트래픽 흐름 (클라이언트 -> 로드밸런서 : 공개IP -> 특정 서버 : 사설IP)
    - 즉, 서버는 클라이언트 접속을 직접 처리하지 않음

- **Q. 로드밸런서는 NAT 이랑 비슷한 역할 같은데 맞음?**
    - 비슷함
        - 결국 둘 다 하나의 IP 로 받고, 내부로 넘기는 장치임
        - NAT : 외부와 내부 연결을 주소 변환을 통해 처리
        - 로드밸런서 : 외부 요청을 받아 내부 서버 중 하나로 전달
    - 근데 본질이 다름
        - NAT
            - 주소 변환이 목적
            - 누구에게 보내야 하는지 이미 정해져 있고, 그 연결이 가능하도록 주소를 바꿔주는 역할
        - 로드밸런서
            - 부하 분산이 목적
            - 여러 서버 중 누구에게 보낼지 결정하는 역할
- **Q. 로드밸런서도 단일 장애점(SPOF) 가 될 수 있는 거 아님? 만일 여러 개 둔다면, 특정 로드밸런서로의 로드밸런싱은 누가 함?**
    - 맞음. SPOF 가 될 수 있음
    - 그래서 로드밸런서도 여러 개 둠
    - 로드밸런서로의 로드밸런싱 처리 방법
        1. DNS 라운드 로빈
            - 클라이언트가 특정 도메인에 접속하려고 할 때, IP 를 라운드 로빈 방식으로 제공
            - 단순하나 특정 로드밸런서가 다운된 것을 감지하지 못한다는 단점이 있음
        2. L4 스위치
            - L4 스위치 : TCP/UDP 수준에서 패킷이나 연결을 보고 어느 뒤쪽 대상에게 보낼지 결정하는 네트워크 장비
            - 이 장비는 SIP, DIP, SPORT, DPORT 등을 정보를 볼 수 있음
                - 이를 기반으로 부하를 분산하는 방식임
            - 즉, 로드밸런서의 로드밸런서임
            - 근데 얘도 SPOF 가 될 수 있음
        3. VIP / VRRP
            - VIP : 가상 IP
            - VRRP : 여러 장비가 하나의 VIP 를 공유할 때, 어떤 장비가 VIP 를 소유할지 정하는 프로토콜
            - 특정 로드밸런서가 VIP 를 소유하다, 장애가 발생하면 다음 로드밸런서가 VIP 를 소유하는 방식으로 동작
                - k8s 의 대표자 선출이랑 비슷
- **Q. 로드밸런서는 특정 서버가 다운된 것을 어떻게 알 수 있음? 헬스 체크는 주기적으로 하니까, 그 사이에 발생한 장애는 캐치 못하는 거 아님?**
    - 맞음. 그래서 헬스 체크 말고도 장치를 둚
    - 이걸 passive health check 라고 함
    - 실시간으로 트래픽을 감시하면서 TCP 연결 실패나, http 응답이 5xx 일 때를 감지하는 것
    - 다만, 이걸 100% 신뢰하는 건 아니고, 헬스 체크와 함께 사용하여 판단함

### 데이터베이스 다중화
- 수평적 규모확장은 데이터베이스에도 적용할 필요가 있음
- 보통 master - slave 관계로 설정
    - CUD 는 master 에서, R 은 slave 에서
        - 다만, 읽기도 master 에서 해야 하는 경우가 존재함. 그래서 실무에서는 이렇게 처리함
            - 일반 조회 → slave
            - 쓰기 직후 조회 / 최신성 중요한 조회 → master
    - **Q. DBMS 가 알아서 처리함?**
        - DBMS 가 알아서 처리하는 건 아니고, 보통 애플리케이션 레벨에서 라우팅 규칙을 둚.
- 대부분 서비스는 읽기 연산이 쓰기 연산보다 비중이 높기 때문에 slave 데이터베이스가 더 많음
- 주 서버가 다운 시, 부 서버가 주 서버로 승격됨
    - 이건 k8s랑 비슷한듯. 합의 통해서 승격시키는 거
    - 이 경우, 부 서버의 데이터가 최신 상태가 아닐 가능성이 존재함
        - 왜냐면 master -> slave 로 복제되는 과정에서 지연이 발생할 수 있기 때문
        - 최신 데이터가 복제되지 않은 상태에서 부 서버가 승격되면, 데이터 정합성에 문제가 생길 수 있음
    - **Q. 어떻게 해결함?**
        - 마스터를 여러 개 두거나, 원형으로 복제하는 방식임
        - 근데 이건 교과서적인 거고, 실무에서는 이렇게 처리함
        1. Semi-Sync Replication
            - master 가 최소 한 개 slave 가 relay log 를 수신한 것을 확인하고 커밋하는 방식
            - 근데 이 방식도 완전하지 않음
                - 최소 한 개 slave 에 데이터가 반영된 걸 확인하기 때문에, 특정 노드에 모든 데이터가 반영되었다고 보장할 수 없음
                - 예를 들어, master 에 a1, a2 가 추가된 경우, s1 에 a1 이, s2 에 a2 가 복제될 수 있음
                - 이 경우, s1 이 승격되면 a2 데이터가 유실됨
            - 그래서 이 방식은 동기와 비동기 그 어딘가에 절충안을 두는 것에 의의를 둚
            - 문제를 해결하기 위해서는 이런 방식을 사용함
                1. GTID 기반 failover
                    - 각 트랜잭션에 id 부여하고, 가장 많은 트랜잭션을 가진 slave 를 승격시킴
                    - 가장 적은 데이터 유실을 보장함
                2. 특정 slave를 우선 ack 대상으로 지정
                    - 데이터 분산을 줄여서, 특정 노드에 모든 데이터가 반영되도록 함
                3. 아에 과반수 노드에 기록한 후 commit
                    - 요즘 사용하는 Consensus 방식임. 3번에서 깊게 다룸
        2. Sync Replication
            - 여러 노드에 동시에 반영한 후 커밋
            - 그러나 이 경우에도 특정 노드에 모든 데이터가 반영되었음을 보장할 수 없음
            - 그래서 리더 장애 시 누구를 리더로 선출해야 하는가? 에 대한 고민이 남음
        3. Consensus
            - 블록체인 비스무리한 방식
            - 과반수가 저장한 것만 commit 하여 공식 기록으로 인정
            - Sync Replication vs Consensus
                - Sync Replication 에서는 리더 장애 시 누구를 리더로 선출해야 하는가? 에 대한 고민이 남음
                - 왜냐면 특정 노드에 모든 최신 데이터가 복제되었음을 보장할 수 없기 때문
                - 반면 Consensus 는 과반수가 저장한 것만 commit 하기 때문에, 장애가 발생하더라도, commit 된 로그를 보존한 노드들만 다음 리더 선출 과정에 참여할 수 있도록 설계할 수 있음
