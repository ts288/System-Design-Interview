## 데이터 센터

- 장애가 없는 상황이라면, 사용자의 요청은 가장 가까운 데이터 센터로 가게 됨
- 이 절차를 지리적 라우팅이라고 함
    - 지리적 라우팅의 GeoDNS 는 사용자의 위치에 따라 가변적으로 IP 주소 변환을 해주는 서비스임
    - **Q. 왜 BGP 사용 안하고 이거 쓰나? CDN 은 왜 BGP 사용하나?**
        - 두 기술이 적용되는 레이어가 다름.
        - GeoDNS 는 사용자가 도메인을 조회할 때 적용되는 기술임
            - HTTP 요청(L7) 이전 단계에서 동작함
            - 사용자 IP 기반으로 가까운 IP 를 반환함
        - BGP 는 IP 가 정해진 상태에서 패킷 라우팅 경로를 결정하는 기술임
            - L3 에서 사용됨
        - **Q. CDN 은 모든 서버에서 같은 IP 를 사용하면서 가장 가까운 데로 라우팅되도록 하는데, 왜 데이터 센터는 각각 다른 IP 를 사용하는가?**
            - CDN 은 상태가 없기 때문에 BGP 에 의존해도 괜찮음
                - 서로 다른 서버에서 데이터를 받아도 된다는 뜻!
                - BGP 가 명시적으로 항상 같은 곳으로의 라우팅을 보장하는 건 아니니까
            - 근데 데이터 센터는 상태가 있기 때문에 명시적으로 라우팅 보장이 되어야 함
                - 그래서 BGP 가 아니라 GeoDNS 를 사용하고, 각각의 IP 를 구분해둔 것!
- 다중 데이터센터 아키텍처를 만들기 위해서 해결해야 할 기술적 난제가 있음
    - 트래픽 우회
        - 올바른 데이터 센터로 트래픽을 보내느 효과적인 방법이 필요함
        - GeoDNS 는 사용자에게서 가장 가까운 데이터센터로 트래픽을 보낼 수 있게 해줌
        - **Q. 데이터센터도 헬스체크 지원하나? 장애 감지는 DNS 가 헬스 체크 날려서 하나?**
            - 지원함
            - 그리고 헬스체크는 DNS 에 붙은 헬스체크 모델이 함
    - 데이터 동기화
        - 데이터센터마다 별도의 데이터베이스를 사용한다면, 각 데이터 베이스 간의 싱크가 맞지 않을 수 있음
        - 이를 해결하기 위해서는 데이터를 여러 데이터 센터에 걸쳐 다중화해야함
        - **Q. 이 때 사용되는 게 데이터 베이스 다중화에 사용되는 방식과 유사한가?**
            - 유사하긴 하나, 스케일과 복잡도가 다름
            - 데이터베이스 다중화는 단일 리전 내에 있기 때문에 네트워크 레이턴시가 수ms 이내라 동기 복제까지 가능
            - 그러나 데이터센터는 네트워크 레이턴시가 수십~수백ms 라, 보통 비동기 복제 사용하고, 일시적 데이터 불일치를 허용함
    - 테스트와 배포
        - 여러 데이터 센터를 사용한다면, 서비스를 여러 위치에서 테스트 해봐야 함
        - 자동화된 배포 도구는 모든 데이터 센터에 동일한 서비스가 설치되도록 함
